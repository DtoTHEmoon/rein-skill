## T01

这是一个典型的"散弹式修改"问题——报价逻辑分散在多处，修了 A 点，B 点的同一逻辑还是旧的。

几个可能的根因：
1. **计算逻辑重复实现**：同一个公式在多个函数里各写了一遍，没有单一来源
2. **缺乏回归测试**：每次改完没有自动验证，靠人眼发现才知道坏了
3. **状态耦合**：某个中间值既被 A 用又被 B 用，改 A 没同步 B

## T05

工具比项目本身更费劲了，这是典型的过度工程化。

问题不在于数量，在于每一层都有摩擦：CLAUDE.md 180行意味着 Claude 每次要消化大量规则，反而容易忽略关键的；25个检查项里可能有一半是边缘情况护栏，实际从没触发过；3层 Multi-Agent 的协调开销可能超过它带来的并行收益。

**最快的诊断方法**：

```
# 看哪些检查项实际在拦截问题
grep -c "FAIL" verify.sh 的历史运行日志
```

建议做一次减法实验：把 CLAUDE.md 压到 50 行以内（只留"不说自明"的规则），把 verify.sh 只保留最近 3 个月真正触发过的检查，看速度有没有变化。

你觉得哪一层最让你感觉在"维护 Harness 而不是做项目"？从那里开始裁。

## T06

**快速定位**：去 [console.anthropic.com](https://console.anthropic.com) 看用量明细，重点看：

- 哪个 API key / 哪类调用占大头
- token 消耗是 input 多还是 output 多

**常见高成本来源**：

1. **Multi-Agent 重复传上下文**：每层都把完整对话历史带过去，token 指数级叠加

3. **180行 CLAUDE.md 每次都进 system prompt**：每个请求都多付这部分 token

## T07

两个独立问题，但根源相同：**Agent 之间没有结构化的信息传递**。

**"对不上"的原因**：需求分析 Agent 的结论以自然语言输出，开发 Agent 收到的是一大段文字，自己重新理解，理解偏了就跑偏了。

**"慢"的原因**：三层串行，每层都带着完整上下文重新启动，协调开销大于并行收益。

---

**最直接的修法**：

需求分析 Agent 输出结构化交付物，而不是自由文本：

```markdown
## 交付物格式
- 功能点列表（编号）
- 每个功能点的验收条件（具体、可测）
- 不做什么（明确排除项）
```

开发 Agent 以这份清单为输入，测试 Agent 以验收条件为测试用例来源。

---

**更值得考虑的问题**：结合【T05】【T06】，这三层是否都必要？需求分析如果是你自己写的，直接给开发 Agent 一份结构化 spec，省掉第一层，速度和准确率可能都会提升。

## T08

这三个问题在这次对话里都出现过，根源是同一件事：**没有让项目知识、规则、和验证机制落地成文件**。

| 症状 | 根源 | 对应修法 |
|------|------|---------|
| 新会话要重新交代背景 | 架构和命名约定只在你脑子里 | 建 CLAUDE.md，`/init` 生成初稿 |
| 修过的 bug 再次出现 | 没有回归测试锁住修复结果 | 每次修 bug 同步补一个测试 |
| API 账单超预期 | 3层 Agent + 180行上下文每次都重复消耗 | 裁剪 Harness，结构化 Agent 间传递 |

三件事按顺序做收益最大：先建 CLAUDE.md（立刻解决上下文问题），再补回归测试（锁住已修的 bug），最后裁 Harness（降成本）。

要从哪个开始？

## T09

短期合理，但有一个条件：**现在就决定好"回头加"的触发点是什么**，否则永远没有"回头"那一天。

结合【T01】——报价 bug 已经复现过一次了。没有验证脚本的情况下继续赶功能，复现概率只会更高，到时候修 bug 的时间会超过现在加验证脚本的时间。

**务实的折中方案**：不用现在写完整的 verify.sh，但每修一个 bug，当场写一个最小测试锁住它。一次 5 分钟，不打断节奏，也不让同一个 bug 第三次出现。
