# 框架详细配置模板

每层包含：定义、启用时机、起步命令、最小可用版本、进阶版本、减法临界点。

架构图：
```
Q1 → Q2 → Q3 ──┐
S1 ─────────────┤→ Q4（统一收口）→ 才算完成
S2 ─────────────┤
S3 ─────────────┘
```

---

## 垂直质量保障层（Q层）

### Q1 规格 SPEC

**解决什么：** AI知道这次要做什么、边界在哪、做完怎么验收。

**缺失症状：** 每次理解不同；做完了但和预期对不上；AI自己发挥填补了没说清楚的部分。

**起步：三个问题，写进CLAUDE.md或任务开头**
```
这次要做什么？（目标）
不做什么？（边界）
做完怎么验证？（验收标准）
```

**好的SPEC的特征：**
- 没有"建议"、"可以"、"推荐"等模糊词
- 有明确的验收条件，不是"做好"而是"能通过XX测试"
- 边界说清楚了，AI不需要猜

---

### Q2 约束层 Rule + Security

> ⚠️ Q2包含两类红线，同级同等重要：
> - 业务红线：AI绝对不能做的业务事项
> - 安全红线：硬编码密钥、.env入库、权限越界
>
> 两类都必须在Q4有对应的检查项，光写规则文字不够。

**解决什么：** AI知道什么绝对不能做，减少反复犯的同类错误，同时防止安全红线被无意突破。

**缺失症状：** 说过的规矩下次又忘；AI自己改了不该改的地方；有硬编码密钥进了代码库。

**起步：从最近3次崩溃出发**

CLAUDE.md业务红线示例：
```markdown
## 绝对禁止
- 禁止修改 .env 文件
- 禁止删除任何已有测试
- 改动部署相关文件前必须先问我

## 必须做
- 每次改完代码必须跑 verify.sh
- 改了价格相关逻辑必须更新单元测试
```

CLAUDE.md安全红线（必须同时加）：
```markdown
## 安全红线
- 禁止API Key、密码硬编码进代码或脚本
- 禁止.env文件提交到Git
- 禁止日志打印用户原始输入
- 修改部署配置前确认端口暴露范围
```

**Rule的原则：**
- 针对反复出现的错误，不是泛泛的"要认真"
- Rule解决"绝对不能"，不解决"怎么做"（那是Q3的事）
- 每条Rule背后要有真实踩坑的原因
- 安全红线和业务红线同级，都不可商量

**减法临界点：** Rule超过15条且AI仍不遵守 → 说明该升级成Q4 Script了，加Rule没用。

### Q3 标准化层 Skill

**解决什么：** 把高频的、步骤固定的动作标准化，不让AI临场发挥。

**缺失症状：** 每次部署步骤不一样；验证方式每次不同；AI自己拼命令总有细节出错。

**什么值得做成Skill：**
- 步骤固定（不需要临场判断）
- 每次都要做（高频）
- 做错了很麻烦（高代价）
- 不希望AI自己发挥

**Skill描述符必须包含反例：**

没有反例时路由准确率：53%；加入反例后：85%，响应时间还降18.1%。
反例不是可选项，是Skill路由能否起作用的关键。

描述符格式：
```
Use when: [具体触发场景]
Don't use when: [明确的排除场景]
Output: [产出物是什么]
```

**Skill文件结构：**
```
项目/.claude/skills/deploy/
├── SKILL.md    # 步骤说明（含反例）
└── scripts/
    └── deploy.sh  # 具体命令
```

**减法临界点：** Skill超过5个且有功能重叠 → 合并，删掉冗余。

### Q4 验证收口层 Scripts ⭐

> ⚠️ Q4不是流程的第四步，而是整个系统的统一出口。
> 不管你用了Q1-Q3还是启用了S1-S3，
> 所有层的改动和产出都必须经过Q4验证才算完成。
> Q4是地基，不是终点——S1/S2/S3建在Q4上面，不是替代它。
> 安全基线和业务验证同等重要，安全问题优先级更高。

**解决什么：** 把"AI说做完了"变成"脚本判定通过了"。建立客观的完成标准。

**Q4必须覆盖四类检查：**
1. 业务验证：核心接口能跑、关键功能正常
2. 安全基线：无硬编码密钥、.env未入库、Dockerfile有USER指令
3. 规范检查：格式违规、禁用语法
4. 基线对比：改动前后对比，杜绝"历史遗留"借口

**最小可用版本（今天就能做）：**
```bash
#!/bin/bash
# verify.sh

# 1. 服务能启动
curl -f http://localhost:8001/health || { echo "❌ 服务启动失败"; exit 1; }

# 2. 安全基线
! grep -rn 'sk-\|password\s*=\s*["'"'"']' --include='*.py' --include='*.ts' \
  --exclude-dir='.git' --exclude-dir='node_modules' . || \
  { echo "❌ 发现硬编码密钥"; exit 1; }
! git ls-files | grep -E '^\.env$' || { echo "❌ .env已入版本库"; exit 1; }

echo "✅ 验证通过"
```

**完整版总验证脚本：**
```bash
#!/bin/bash
PASS=0
FAIL=0

check() {
    local name=$1
    local cmd=$2
    if eval "$cmd" > /dev/null 2>&1; then
        echo "✅ $name"
        ((PASS++))
    else
        echo "❌ $name"
        ((FAIL++))
    fi
}

echo "=== 业务验证 ==="
check "服务启动" "curl -sf http://localhost:8001/health"
check "核心接口" "curl -sf http://localhost:8001/核心接口"

echo "=== 安全基线 ==="
check "无硬编码密钥" "! grep -rn 'sk-\|password\s*=\s*['\''\"']' \
  --include='*.py' --include='*.ts' \
  --exclude-dir='.git' --exclude-dir='node_modules' ."
check ".env未入版本库" "! git ls-files | grep -E '^\.env$'"
check "Dockerfile有USER指令" \
  "! find . -name 'Dockerfile' | xargs grep -L 'USER' 2>/dev/null | grep ."

echo "=== 规范检查 ==="
check "单元测试通过" "python -m pytest tests/ -q"

echo "=== 结果 ==="
echo "通过: $PASS / 失败: $FAIL"
[ $FAIL -eq 0 ] || exit 1
```

**基线对比用法：**
```bash
# 改动前
bash verify.sh > baseline.log
# 做改动...
# 改动后
bash verify.sh > current.log
# 对比
diff baseline.log current.log
```

**Rule配合使用：**
```markdown
# CLAUDE.md
## 绝对禁止
- 任何代码改动完成后，必须运行 bash verify.sh 通过才算完成
- verify.sh 不通过，不允许说"已完成"
```

**减法临界点：** 检查项超过20条且有明显从未触发过的项 → 删掉，保持精简。

## 水平规模扩展层（S层）

### S1 上下文管理 Context

**解决什么：** 防Context Rot——会话越长，关键信号越容易被噪声稀释，AI开始遗忘同一会话里的早期决策。

**启用时机：**
- 同一会话到第15-20轮开始出现前后矛盾
- AI忘记同一会话里刚做过的决策
- API成本异常上涨但工作量没有明显增加

**分层管理策略：**
- 常驻层：身份定义、项目约定、绝对禁止项（短、硬、可执行）
- 按需加载：Skills和领域知识（描述符常驻，内容触发时注入）
- 运行时注入：当前时间、渠道ID等动态信息
- 记忆层：跨会话经验写入MEMORY.md，需要时读取

**起步建议：**

在dev-map.md里加"已定决策"区块：
```markdown
## 已定决策（不再讨论）
- 价格计算统一走pricer.py，不在前端算
- 数据库用PostgreSQL，不换
```
每次会话开始时把这个文件丢给AI。

**S1也要过Q4：**
S1相关文件（MEMORY.md、dev-map的决策区块）有更新时，
在Q4加一个检查项确认关键决策文件存在且格式正确。

**减法临界点：** MEMORY.md超过100条且AI开始忽略 → 精简到最关键的20条。

### S2 知识库 dev-map + Memory

**解决什么：** 让AI在长期项目里不重复造轮子，知道整个项目的结构和惯例。

**启用时机：**
- 项目持续迭代超过2个月
- AI开始重复造轮子，不知道已有实现
- 旧设计被新设计莫名其妙冲掉

**S2也要过Q4：**
dev-map记录的是"AI应该知道的项目结构"，
更新dev-map后要在Q4验证记录的路径和模块实际存在：
```bash
check "dev-map记录的核心模块存在" "test -f src/核心模块.py"
```

**dev-map vs CLAUDE.md：**
- CLAUDE.md：规则和禁止（AI必须遵守的）
- dev-map：项目地图（AI应该了解的）

**最小可用版本：**
```markdown
# dev-map.md

## 项目结构
- 报价逻辑在 ark-engine/rules.py
- 前端API代理在 ark-platform/app/api/
- 数据库操作在 lib/db.ts

## 已有实现，不要重复造
- Excel生成：用 openpyxl，不要用其他库
- 价格计算：统一走 pricer.py，不要在别处算

## 惯例
- 新增接口必须在 /health 接口里更新版本号
- 部署前必须跑 verify.sh
```

**减法临界点：** dev-map和实际代码结构脱节超过1个月 → 说明没人维护，考虑删掉或大幅精简。

---

### S3 分工 Multi-Agent

**解决什么：** 把长链路任务拆成角色分工，防止单Agent自说自话。

**启用时机（必须同时满足三个条件）：**
1. 单Agent在长链路任务里明显失稳、前后矛盾
2. Q4已经是真正的统一收口（没有Q4就没有客观判断多Agent做完没有的标准）
3. 任务链路超过 需求→方案→开发→验证 四个阶段

**S3的前置条件：Q4必须先到位**

Multi-Agent的每个Agent产出都要经过Q4验证，
没有Q4收口的Multi-Agent，协作会更乱而不是更清晰。

**什么时候不需要（大多数情况）：**
- 单人项目，任务链路不超过3步
- 加了Multi-Agent反而比单Agent慢
- Q4还没有真正到位

**最小分工：先拆两层**
```
需求分析Agent → 实现Agent
（不要一上来就搭5个Agent）
```

**减法临界点：** S3跑起来比单Agent更慢或更乱 → 退回单Agent，重新评估。
