## 案例001：内部AI报价系统（单人非技术背景）

**项目画像**
- 类型：内部业务自动化工具
- 功能：体检报价自动生成，替代人工查表
- 团队：1人主导，非技术背景，使用Claude Code
- 周期：3个月从0到生产
- 技术栈：FastAPI + Next.js + PostgreSQL + Docker

**五维度**
- 容错成本：中（内部使用，出错影响营销人员效率）
- 迭代频率：高（每周有新需求）
- 角色分工：单人
- 任务链路：中（规则引擎+Excel生成，有完整链路）
- 初始状态：零配置，从Dify平台迁移

**最终Harness配置（达到L4）**

```
L1 规格：CLAUDE.md写清楚项目背景、两个模块的业务规则边界
L2 规则：
  - 禁止修改价格计算核心逻辑（pricer.py）不经测试
  - 禁止直接在服务器修改文件，必须走打包部署流程
  - 改动ark-engine必须先更新ark-platform，顺序不能反
  - 中文文件名必须URL encode，禁止直接塞进HTTP header
L3 技能：
  - 部署Skill（打包→传输→验证三步标准化）
  - Excel生成Skill（openpyxl规范用法）
L4 验证：
  - verify.sh：health check + 核心接口响应 + 因素数量校验
  - 部署前后基线对比
L5：不需要，单人项目
L6：CLAUDE.md里有简单的文件结构说明，未做完整dev-map
```

**关键踩坑（Harness补充的来源）**

| 踩坑 | 根因 | 加入的Rule/Script |
|------|------|-----------------|
| 部署后代码没更新 | 没有验证步骤，靠手动curl | 加verify.sh，改完必须跑 |
| 中文文件名500报错 | 没有编码规范Rule | 加Rule：中文header必须URL encode |
| Docker镜像体积翻倍 | .dockerignore缺少*.tar.gz | 加Rule+.dockerignore更新 |
| 前后端部署顺序搞反 | 没有部署顺序Rule | 加Rule：先后端后前端，铁律 |
| 环境变量改了不生效 | pm2 restart不重载env | 加Rule：改.env必须重新build |

**效果对比**
- 报价时间：30分钟以上 → 3分钟以内
- 部署失误率：平均每次部署1-2个问题 → 接近零
- 新功能开发：每次都要重新解释背景 → CLAUDE.md保障上下文
